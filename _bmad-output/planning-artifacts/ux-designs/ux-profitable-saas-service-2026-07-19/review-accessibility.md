# Accessibility Review — Algerian B2B Lead Platform V1

Scope: accessibility + i18n/RTL behavioral risk only. Reviewed against WCAG 2.1 AA (public) / A + AA-on-interactive (app), the trilingual AR(RTL)/FR/EN model, and the low-end Android + 4G reality.

## Overall verdict

The RTL/SR foundation is unusually strong for a spine-stage artifact — the aria-disabled primitive, bidi-isolation rule, icon-mirroring rule, logical-property hard rule, and scoped aria-live for credits/toasts are all correct and load-bearing. But the spec goes silent exactly where its own AA claims bite: the dir-flip focus lifecycle, all four auth forms, per-fragment `lang` for mixed-script data, and two token-level decisions (footer link pair, 0.6-opacity actionable control) that mandate AA failures as written. All findings are fixable in one revision pass; nothing is architectural.

## Findings

- **[critical]** Locale switch has no focus-preservation or announcement rule. The preserved-state lists ("filters, keyword, page/cursor, expanded reveal rows, scroll anchor") omit `document.activeElement`, and nothing announces the locale change itself. A keyboard/SR user mid-session (the designed UJ-3 flow) can be dumped to `<body>` on a re-rendered, re-flowed RTL↔LTR tree with no indication the switch succeeded. (EXPERIENCE.md — Component Patterns/Locale switcher; Trilingual & RTL Interaction Model §4; Key Flows UJ-3 step 3). *Fix:* add to the switch contract: "focus is captured pre-flip and restored to the same logical control post-flip (by stable id, not DOM node); if the node is re-created, focus moves to the locale switcher trigger; an `aria-live="polite"` visually-hidden status announces 'Interface language: Français' in the new locale."
- **[high]** Forms are claimed AA but zero form-a11y behavior is specified. /signup, /login, /verify-email, /password-reset have no label-association rule, no `autocomplete` attributes (`email`, `current-password`, `new-password`), no error-identification/association spec (`aria-invalid` + `aria-describedby`, inline vs summary), no error-message direction rule in RTL, and no localized validation strings — the microcopy table contains no form errors at all. Password rules communication is absent. (EXPERIENCE.md — Information Architecture auth rows; Accessibility Floor; Voice and Tone table). *Fix:* add a Forms subsection to Accessibility Floor covering labels-always (never placeholder-as-label), autocomplete tokens, per-field inline errors with `aria-describedby` + an error summary on submit, errors rendered in the active locale and `dir`, and password requirements stated adjacent to the field pre-submit; add validation strings to the trilingual microcopy table.
- **[high]** Footer link pair fails AA as tokened: `muted-foreground` (#64748B) on `muted` (#F1F5F9) computes ≈4.35:1, below the 4.5:1 required for the 14px (`typography.small`) links — and this pair is absent from the verified contrast table. The footer (and its locale switcher) sits on every public AA-target page. (DESIGN.md — Components/footer; Colors contrast table; EXPERIENCE.md — Accessibility Floor public-AA rule). *Fix:* use `foreground` or a darker slate for footer links, or lighten the footer fill; add `muted-foreground on muted` to the verified-pairs table with an explicit verdict.
- **[high]** `disabled-opacity: 0.6` on the reveal button renders an *actionable* `aria-disabled` control's label at ≈2.6:1 (white on 60%-opacity teal over white). WCAG's inactive-component contrast exemption does not apply — this control is focusable and click-opens the recovery dialog, i.e., it is an active conversion path that low-vision users must be able to read. (DESIGN.md — components/reveal-button token + Components/reveal-button aria-disabled row). *Fix:* drop the opacity fade for the aria-disabled state; use a tonal treatment (e.g., `muted` fill / `foreground` label / 1px `border`) that holds 4.5:1, keeping opacity only if a non-opacity style already signals state.
- **[high]** Rate-limited search state says "Apply disabled until 00:00 Africa/Algiers" — unqualified `disabled`, contradicting the spine's own Disabled-but-actionable primitive (rate limiting is metering). If implemented as native `disabled`, the button becomes unfocusable and its explanation (the come-back-tomorrow message) has no specified association to the control. (EXPERIENCE.md — State Patterns/rate-limited (search) vs Interaction Primitives/Disabled-but-actionable). *Fix:* restate as `aria-disabled` with the rate-limit message rendered inline adjacent to Apply and referenced via `aria-describedby`, or keep Apply enabled and show the refusal on activation.
- **[medium]** Tooltip touch reachability is unspecified. The aria-disabled pattern guarantees *keyboard* reachability ("focus triggers the same explanation hover does") but says nothing for touch: the free-tier xlsx button, the ≤10-credit warning pill tooltip, and the 0-credit reveal tooltip have no tap behavior defined (tap = tooltip? tap = recovery dialog directly?). The Upgrade dialog row lists "xlsx tooltip" as an entry point without defining how a touch user opens it. (EXPERIENCE.md — Accessibility Floor/aria-disabled; Component Patterns/Export modal, Credits pill, Upgrade dialog). *Fix:* specify first-tap-shows-tooltip / second-tap-activates (or tap-opens-dialog-with-explanation-inline) for every tooltip-bearing control, and make the warning-pill explanation reachable on the /credits destination it links to.
- **[medium]** Touch-target floor contradicts fixed component heights. EXPERIENCE mandates ≥44px on mobile "including chip remove buttons, combobox options, and the bottom-sheet handle," but DESIGN fixes heights with no mobile/desktop distinction: credits-pill 28px, filter-chip 28px, reveal-button 32px, locale-switcher 36px, persistent banner 40px. A 44px remove button inside a 28px chip is impossible; nothing says which surfaces inflate. (EXPERIENCE.md — Accessibility Floor + Responsive & Platform; DESIGN.md — component height tokens). *Fix:* state the reconciliation rule — e.g., "named heights are desktop (`≥md`); on `<md`, all interactive targets inflate to ≥44px via padding while visual height is preserved" (or raise the heights).
- **[medium]** aria-live coverage has gaps on state changes that currently rely on vision. (a) Chargily status card: the polling→success flip gets a toast, but the card state change itself and the timeout degradation ("payment received, credits post soon") have no `role="status"`/live spec. (b) Checklist "live check-off" is visual-only — SR users never learn a step completed. (c) Reveal success: the button has no `aria-expanded`/`aria-controls` spec, so a SR user activating Reveal gets only the pill decrement, not "contact revealed" or where the new content is. (EXPERIENCE.md — Component Patterns/status card, checklist card, Reveal control; State Patterns/webhook-pending). *Fix:* put `role="status"` on the status card, announce checklist completions via the existing polite region, and spec `aria-expanded` + `aria-controls` on the reveal control with the expanded region labeled.
- **[medium]** Per-fragment `lang`/`dir` is only spec'd for fall-through *UI strings*, not for mixed-script *data*. Arabic-script content appears inside Latin locales by design — wilaya fallback names in the combobox (`31 — وهران`), the wilaya column in results, the 3-language /wilayas table cells, Arabic proper nouns in FR/EN UI — with only a *font* rule, no `lang="ar"`/`dir="rtl"` fragment rule, so screen readers will read Arabic with a French/English TTS voice. (EXPERIENCE.md — Accessibility Floor/Arabic screen readers; Component Patterns/Wilaya combobox; Trust Surfaces//wilayas; DESIGN.md — Typography/Stacks). *Fix:* extend the mixed-locale rule to data: "any Arabic-script fragment carries `lang="ar" dir="rtl"` on its own element regardless of source (string fall-through or data), including combobox options, table cells, and CSV-preview rows."
- **[medium]** Modal focus management is never stated — no focus-trap, initial-focus, or focus-return rule for the export modal, upgrade dialog, or recovery dialog, including the specific case where focus must return to an `aria-disabled` trigger (which must therefore remain in the DOM). Other inherited Radix behaviors (Select keyboard model) are restated, so this omission reads as unowned. (EXPERIENCE.md — Component Patterns/Export modal, Upgrade dialog, Reveal control; Interaction Primitives). *Fix:* add one line to Accessibility Floor: "all dialogs trap focus, set initial focus to the first focusable element (or the destructive-safe control), and return focus to the invoking control on close — including aria-disabled invokers."
- **[medium]** /wilayas — a public AA page whose entire payload is a 58-row data table — has no table-semantics spec: no `<caption>`, no `th scope="col"` per language column / `scope="row"` on the code, and the client-side filter input has no label rule. (EXPERIENCE.md — Trust Surfaces//wilayas). *Fix:* require caption ("58 wilayas of Algeria — codes 1–58 with Arabic, French, and English names"), scoped headers, a labeled filter input, and per-cell `lang` (see per-fragment finding).
- **[medium]** No bypass-blocks mechanism (skip link) anywhere. With a sticky header on every page and a persistent filter sidebar before results in DOM order, keyboard users tab through full chrome per page; public AA pages have the same issue. (EXPERIENCE.md — Information Architecture/Persistent chrome; Accessibility Floor). *Fix:* spec a skip-to-content link (first focusable element, visible on focus) on all pages, plus skip-to-results on search surfaces.
- **[medium]** Sortable results table and company row-click lack SR/keyboard semantics: "Sortable… with a chevron" has no `aria-sort` or sort-change announcement spec, and "Company row click → /companies/:id" defines no keyboard path (a clickable row is not focusable by default). (EXPERIENCE.md — Component Patterns/Results table; DESIGN.md — Components/results-table). *Fix:* require `aria-sort` on the active header + polite announcement of sort change; make the company name a real link inside the row.
- **[low]** Warning pill (≤10 credits) encodes state by container-color swap plus an *on-demand* tooltip only — no persistent non-color signal (icon delta or text). Color-blind users get nothing until they somehow trigger the tooltip (which touch users can't — see tooltip finding). (DESIGN.md — Components/credits-pill; EXPERIENCE.md — State Patterns/low-credit). *Fix:* add a persistent warning icon (lucide `alert-triangle`) beside the balance in the warning state.
- **[low]** Reduced-motion rule covers only the locale-switch text fade ("no other animation is load-bearing" is a semantics statement, not a motion rule): the ≤200ms pill color transitions and the reveal/status-card spinners have no `prefers-reduced-motion` treatment specified. (EXPERIENCE.md — Accessibility Floor/Reduced motion; DESIGN.md — Components/credits-pill, reveal-button, status-card). *Fix:* extend the rule: "under `prefers-reduced-motion`, color transitions are instant and spinners are replaced by a static busy indicator + text."
- **[low]** Mobile bottom-sheet dismiss behavior is unspecified — no swipe-to-dismiss, scrim-tap, or close-button rule, and no focus-return to the Filters(n) trigger on close. (EXPERIENCE.md — Information Architecture/Filter sidebar; Responsive & Platform). *Fix:* spec visible close button + scrim-tap + swipe-down dismissal, Esc on hybrid devices, and focus return to the trigger.
- **[low]** Founder-headshot deferral has no alt-text rule — neither for the future headshot (informative: "Akram, founder") nor for the interim `muted` placeholder block (should be decorative, `alt=""`/aria-hidden). (DESIGN.md — Brand & Style/deferred assets; EXPERIENCE.md — Trust Surfaces//about). *Fix:* add the alt rule now so it survives the asset landing pre-launch.
- **[low]** Wilaya combobox keyboard model is thin: "Keyboard-active option: muted fill" is styling only — the multi-select toggle key, removal of selected chips *inside the trigger* (results-row chips are covered, trigger chips are not), and Esc/focus-return are unspecified; the internal search input risks placeholder-as-label. (DESIGN.md — Components/wilaya-combobox; EXPERIENCE.md — Component Patterns/Wilaya combobox). *Fix:* spec Enter/Space toggles selection, Backspace/Delete or per-chip remove buttons reachable in the trigger's tab order, labeled search input (`aria-label` or visible label), Esc closes and returns focus.
- **[low]** Public content pages claim "crawlable semantic HTML" but no heading-hierarchy rule (single h1, ordered h2/h3) exists for the trust pages that carry the SEO+AA burden. (EXPERIENCE.md — Trust Surfaces). *Fix:* one line in Trust Surfaces or Accessibility Floor mandating valid heading order per page.

## What passes

- **aria-disabled primitive is correct and central**: focusable blocked actions, keyboard-reachable tooltips, recovery-path activation instead of dead-ends; reveal explicitly "click/Enter opens the recovery dialog" (EXPERIENCE.md — Interaction Primitives; Accessibility Floor).
- **Locale switcher keyboard model fully spec'd** (open, arrow, Enter, Esc) in both header and footer instances (EXPERIENCE.md — Accessibility Floor).
- **aria-live scoped correctly**: `polite` for credit balance and toasts, assertive reserved for payment failure — exactly the right escalation discipline (EXPERIENCE.md — Accessibility Floor; Credits pill).
- **Bidi isolation with mechanism** (`<bdi>` / `unicode-bidi: isolate`) for numerals, phones, codes inside RTL text; Western-numerals hard rule (0-9) in all locales removes a whole class of bidi/numeral bugs (EXPERIENCE.md — Accessibility Floor; DESIGN.md — Typography/Numeral rule).
- **Icon-mirroring rule is precise**: directional icons (back arrows, in-flow chevrons) mirror, non-directional (globe, download) never (EXPERIENCE.md — Accessibility Floor).
- **Focus order follows visual order in both directions; 2px ring + offset "must survive RTL mirroring unchanged"** (EXPERIENCE.md — Accessibility Floor; DESIGN.md — Components intro).
- **Root `lang="ar" dir="rtl"` + mixed-locale fragment `lang` rule** exists for fall-through strings (needs extension to data — see findings) (EXPERIENCE.md — Accessibility Floor).
- **Logical-properties hard rule with forbidden-list**, instant `dir` flip, no layout animation or skeleton interstitials on switch — good for vestibular users and low-end devices (DESIGN.md — Layout & Spacing; EXPERIENCE.md — Foundation).
- **Contrast table verifies 13 load-bearing pairs** including all four semantic container/on-container pairs, the warning-pill pair, the danger-banner pair, and the best-value badge pair; "brand overrides must hold ratios" constrains the placeholder-palette swap (DESIGN.md — Colors; EXPERIENCE.md — Accessibility Floor).
- **`prefers-reduced-motion` disables the locale text fade** (EXPERIENCE.md — Accessibility Floor).
- **No infinite scroll (banned), pagination only; modal stacks deeper than one banned** — both focus- and SR-friendly (EXPERIENCE.md — Interaction Primitives/Banned).
- **No placeholder-as-label in filter groups** (explicit group labels), no flags for languages, checklist avoids strikethrough for Arabic legibility, watermark treated as content not overlay, already-revealed auto-visible (no re-interaction burden) (DESIGN.md — Components; EXPERIENCE.md — Component Patterns).
- **Touch-target ≥44px floor exists with named inclusions** (chip remove, combobox options, sheet handle) — needs reconciliation with fixed heights (see findings) (EXPERIENCE.md — Accessibility Floor).
